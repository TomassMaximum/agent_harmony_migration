# 鸿蒙迁移 Agent 开发路线图

## 1. 文档目标

本文档用于把当前项目从“聊天式原型”推进到“具备分阶段迁移闭环的 MVP”。

当前代码已经具备以下基础能力：

- `AgentLoop` 单步驱动 LLM 与工具调用
- 本地工具体系：`list_dir`、`read_file`、`search_text`、`run_command` 等
- `session/chat` 级别记忆持久化
- CLI 交互入口与 OpenAI 风格 Web 适配层

当前主要缺口：

- 缺少结构化工程产物存储，仍以 chat 记忆为主
- 缺少真正的阶段编排器（orchestrator）
- 缺少 analyze/design/implement/review 四阶段约束
- 缺少 Harness 与测试验收闭环
- 现有入口和主流程之间仍有未收口断点

## 2. 总体路线

### 里程碑 1：原型收口，形成稳定底座

目标：
把当前“能跑但不够稳定”的原型收口成一个可靠的代理底座，解决入口断点、事件协议不一致、测试缺失、权限策略临时化等问题。

交付：

- 修复 CLI 主入口断点
- 统一 `AgentLoop` 的执行语义和返回结构
- 明确 CLI / 交互式 CLI / Web 入口职责
- 为核心基础模块补最小回归测试
- 把权限控制从临时放开状态收口到可控状态

阶段完成标准：

- 能通过 CLI 发起一次稳定的只读分析任务
- 能通过交互式 CLI 恢复历史 session 并继续执行
- 能通过 Web 适配层完成一次标准问答
- 工具调用、事件输出、最终答复的行为一致
- 不存在“脚本能启动但调用缺失方法”的问题

### 里程碑 2：落地结构化工程记忆

目标：
从“记住对话”升级到“记住工程状态”，引入 `Artifact Store`。

交付：

- 新增 `project_memory/` 目录结构
- 定义并实现 `project_overview.json`
- 定义并实现 `design.json / implementation.json / review.json` 的 schema
- 实现结构化产物的读取、写入、恢复与版本管理

阶段完成标准：

- Analyze 阶段能稳定产出 `project_overview.json`
- 系统重启后可以从结构化产物恢复状态
- 后续阶段不再主要依赖 chat 历史理解工程

### 里程碑 3：实现分阶段 orchestrator

目标：
把当前通用聊天代理改造成流程化迁移系统。

交付：

- 新增 orchestrator 控制阶段流转
- 实现 `analyze_mode / design_mode / implement_mode / review_mode`
- 加入阶段级只读/可写约束
- 引入单模块推进机制

阶段完成标准：

- 系统能按阶段推进，而不是让模型自由决定流程
- 每个阶段都有固定输入、输出和限制
- 同一时刻只有一个模块进入实现阶段

### 里程碑 4：补齐 Harness 与验收闭环

目标：
形成最小可用的迁移闭环，而不是停留在分析和改代码层面。

交付：

- 定义 Contract Test 与 Module Test 接口
- 引入测试执行与结构化测试结果
- 实现 review 产物与决策逻辑
- 支持 `approved / rework / redesign`

阶段完成标准：

- 单模块可以完成 design -> implement -> test -> review 全链路
- review 决策可以驱动下一步动作
- 能形成可落盘、可复现的验收结果

## 3. 推荐执行顺序

建议按以下顺序推进：

1. 先完成里程碑 1，保证当前底座可靠
2. 再完成结构化产物层，避免后续继续堆 chat 逻辑
3. 然后引入 orchestrator，正式从“聊天”转向“流程”
4. 最后补 Harness 和验收闭环，形成真正 MVP

## 4. 里程碑 1 详细拆解

里程碑 1 的目标不是增加更多能力，而是收口当前已有能力，降低后续继续开发的风险。

### M1-T1：修复主入口断点 `[已完成]`

任务目标：
修复 `scripts/run_agent.py` 与 `AgentLoop` 之间的断裂，确保“命令行主入口”可以独立完成一次完整任务。

当前问题：

- `scripts/run_agent.py` 调用了 `agent.run(args.task)`，但 `AgentLoop` 中没有 `run()` 方法
- 主入口语义不清晰，不知道是“自动执行到 final”还是“只启动 session”

要实现的功能：

- 为 `AgentLoop` 增加明确的高层运行接口，或者改造 `scripts/run_agent.py` 改用现有 `start_session + step_once` 机制
- 明确主入口的执行模型：
  - 启动 session
  - 注入初始任务
  - 循环执行 step
  - 遇到 final 时停止
  - 超过最大步数时给出明确退出结果
- 对异常进行统一处理，避免入口脚本直接崩溃

建议输出：

- 一个可直接运行的 CLI 主入口
- 清晰的返回语义：成功 final、达到步数上限、执行异常

完成标准：

- `scripts/run_agent.py` 可以独立运行
- 给定简单分析任务时，脚本能输出最终结果
- 最大步数命中时，行为可预测

#### M1-T1 设计目标进一步收敛

当前状态：

- 已完成
- 已实现统一高层执行接口 `run_until_stop()`
- `run_agent.py` 已改为可运行的薄入口
- `chat_agent.py` 已切到统一执行主干
- `openai_adapter.py` 的非流式路径已切到统一执行主干
- 运行产物 `task.txt` 已删除，CLI 不再依赖文件注入初始任务

这一任务块只解决“主入口断点”和“主入口运行模型缺失”两个问题，不处理更广义的执行语义统一。也就是说：

- `M1-T1` 只要求 `run_agent.py` 可用
- `M1-T1` 不要求一次性彻底重构所有入口
- `M1-T1` 产出的能力应能被 `M1-T2` 继续复用，而不是推倒重来

建议采用的实现方向：

- 保持 `scripts/run_agent.py` 尽量薄
- 把“一次性任务执行”主循环下沉到 `AgentLoop`
- `run_agent.py` 只负责参数解析、实例化 agent、打印结果、设置退出码

这样做的原因：

- 避免主循环散落在多个入口脚本中
- 后续 `chat_agent.py` 和 `openai_adapter.py` 可以复用相同的高层接口
- 让 `AgentLoop` 真正成为执行核心，而不只是单步容器

#### M1-T1 子任务拆分

##### M1-T1-A：定义主入口的目标行为 `[已完成]`

任务目标：
先把“一次性 CLI 任务”到底要怎么运行说清楚，避免修了断点但行为仍不稳定。

要实现的功能：

- 定义 `run_agent.py` 的标准行为：
  - 接收一个任务字符串
  - 创建一个新 session
  - 将任务作为初始输入启动 agent
  - 自动循环执行多个 step
  - 在出现 final 时停止
  - 在达到最大步数时停止
  - 在出现系统异常时停止
- 定义它与交互式入口的区别：
  - 不支持中途暂停
  - 不等待用户插入补充输入
  - 以“一次跑完一轮任务”为目标
- 定义命令行输出层级：
  - 是否打印 thought / tool trace
  - 是否只打印 final answer
  - 错误信息打印到 stdout 还是 stderr

建议输出：

- 一份简短但明确的“主入口行为规范”
- 对应到代码里就是一个高层方法的行为注释或文档说明

完成标准：

- 团队成员能明确理解 `run_agent.py` 是“批处理入口”而不是“交互入口”
- 后续编码时不会再出现不同人理解不同的情况

##### M1-T1-B：补齐 AgentLoop 的高层运行接口 `[已完成]`

任务目标：
为 `AgentLoop` 增加一个真正可供主入口调用的高层接口，消除当前 `agent.run()` 缺失的问题。

当前缺口：

- `AgentLoop` 当前只有：
  - `start_session()`
  - `step_once()`
  - `send_user_message()`
- 缺少“从启动到收敛”的单轮批处理接口

要实现的功能：

- 新增一个高层运行方法，职责至少包含：
  - 调用 `start_session()`
  - 循环执行 `step_once()`
  - 收集事件
  - 判断停止条件
  - 返回统一结果
- 该方法需要支持：
  - 初始任务输入
  - 最大步数限制
  - 是否注入 chat memory
  - 是否打印 trace
- 该方法应避免把展示逻辑耦合进核心流程

建议接口方向：

- 方案 A：新增 `run_task(task: str, ...) -> RunResult`
- 方案 B：新增兼容方法 `run(task: str) -> str`，内部再调用更底层的统一运行接口

实际落地：

- 未采用 `run_task()` / `run()` 命名
- 最终统一为 `run_until_stop()`
- 该命名直接表达行为：驱动当前 session 持续执行直到停止

原因：

- `run_task()` 语义比 `run()` 更清晰
- 返回结构化结果更利于后续被 Web/CLI 共同复用
- 可以避免过早把接口锁死成“只能返回字符串”

涉及文件：

- `agent/loop.py`

完成标准：

- `AgentLoop` 自身具备“跑完整一轮任务”的能力
- 主入口不需要自己手写 step 循环
- 该高层接口的职责边界清晰

##### M1-T1-C：定义高层运行结果对象 `[已完成]`

任务目标：
不要让 `run_agent.py` 依赖“某个方法刚好返回字符串”，而是先定义清晰的运行结果。

要实现的功能：

- 定义一个最小运行结果结构，建议至少包含：
  - `final_answer`
  - `stop_reason`
  - `step_count`
  - `events`
  - `session_id`
  - `chat_id`
- 定义 `stop_reason` 的最小范围：
  - `final`
  - `max_steps`
  - `error`
- 明确当未拿到 final 时，`final_answer` 如何填充：
  - 空字符串
  - 默认提示文本
  - 由入口层决定兜底文案

建议输出：

- 一个 dataclass 或等价结构

原因：

- 主入口只需要关注结果，不应反向解析执行过程
- 这也是 `M1-T2` 统一执行语义的前置基础

涉及文件：

- `agent/loop.py`
- 可能新增到 `agent/custom_types.py` 或新建结果类型文件

完成标准：

- 主入口可以基于结构化结果做输出与退出码判断
- 后续其他入口也可复用该结果对象

##### M1-T1-D：实现主循环停止条件 `[已完成]`

任务目标：
把“一次性任务什么时候停”做成清晰、稳定、可测试的逻辑。

要实现的功能：

- 在高层运行接口中实现主循环
- 循环内处理以下停止条件：
  - `step_once()` 产生 final
  - 已达到 `max_steps`
  - 发生未处理异常
- 对每个停止条件写清楚行为：
  - 是否保存 session
  - 是否生成摘要
  - 返回什么 `stop_reason`
  - 主入口是否视为失败退出

建议规则：

- `final`：
  - 正常保存 session
  - 正常生成摘要
  - 返回成功
- `max_steps`：
  - 仍保存 session
  - 可选择生成摘要
  - 返回非 final 结束
- `error`：
  - 尽量保留当前上下文并保存 session
  - 返回错误停止

涉及文件：

- `agent/loop.py`

完成标准：

- 相同输入下，停止行为稳定可预期
- 不会出现无限循环
- 达到步数上限时不会伪装成成功完成

##### M1-T1-E：改造 run_agent.py 为薄入口 `[已完成]`

任务目标：
让 `run_agent.py` 不再持有业务流程，只做脚本入口职责。

要实现的功能：

- 保留参数解析：
  - `task`
  - `model`
  - `max-steps`
- 实例化 `AgentLoop`
- 调用新的高层运行接口
- 根据结果做输出：
  - 正常 final 时打印最终答复
  - 命中步数上限时打印说明
  - 异常时打印错误
- 根据结果设置退出码

建议退出码策略：

- `0`：成功拿到 final
- `1`：系统异常
- 可选 `2`：达到最大步数但未收敛

原因：

- 退出码会直接影响脚本在自动化环境中的可用性
- 后续如果要接 CI 或外部调度，这一点很重要

涉及文件：

- `scripts/run_agent.py`

完成标准：

- `run_agent.py` 只保留入口层代码
- 没有重复的 step 循环和事件解析逻辑
- 输出和退出码规则一致

##### M1-T1-F：定义最小输出格式 `[已完成]`

任务目标：
确保命令行输出适合人读，也适合后续脚本消费。

要实现的功能：

- 明确默认输出内容：
  - 是否显示 `===== FINAL ANSWER =====`
  - 是否显示 stop reason
  - 是否显示 step count
- 明确错误输出位置：
  - 用户结果输出到 stdout
  - 错误和调试输出到 stderr
- 明确当没有 final 时的展示规范

建议默认行为：

- stdout：
  - 输出最终结果主体
- stderr：
  - 输出错误、调试和运行状态

可选增强：

- 后续加入 `--json` 或 `--trace` 参数，但不作为 `M1-T1` 必选项

涉及文件：

- `scripts/run_agent.py`
- 如需要，可补辅助格式化函数

完成标准：

- 人工运行时输出清晰
- 后续 shell 脚本可以稳定消费 stdout
- 不同停止原因有可区分的展示

##### M1-T1-G：补最小验证场景 `[已完成]`

任务目标：
让 `M1-T1` 完成后可以被快速验证，而不是只靠目测代码。

要实现的功能：

- 设计 3 个最小验证场景：
  - 场景 1：简单只读任务，成功拿到 final
  - 场景 2：模型持续不 final，触发 `max_steps`
  - 场景 3：模型调用异常或返回非法 JSON，触发错误停止
- 为每个场景定义预期：
  - stdout 应该看到什么
  - stderr 应该看到什么
  - 退出码是多少
  - session 是否被保存

实际完成情况：

- 已完成 `final / max_steps / error` 三条停止路径的 smoke 验证
- 已完成 `run_until_stop()` 的 step callback 验证
- 已完成 `py_compile` 级别静态校验

涉及文件：

- 先更新文档
- 后续可扩展到测试文件

完成标准：

- `M1-T1` 完成后可以用统一方法验收
- 不需要人工反复阅读源码判断是否完成

#### M1-T1 文件级改造建议

建议优先改造以下文件：

- `agent/loop.py`
  - 新增高层运行接口
  - 收口主循环与停止条件
  - 定义并返回结构化运行结果
- `scripts/run_agent.py`
  - 改为薄入口
  - 接入新的高层运行接口
  - 统一输出与退出码
- `agent/custom_types.py` 或新文件
  - 视情况新增运行结果类型

尽量不要在 `M1-T1` 里同步大改以下文件：

- `scripts/chat_agent.py`
- `scripts/openai_adapter.py`

原因：

- `M1-T1` 的目标是先把主入口补齐
- Web 和交互入口的统一改造应留给 `M1-T2` 和 `M1-T3`
- 这样可以控制改单范围，避免第一个任务块膨胀

#### M1-T1 风险点

- 如果高层运行接口直接返回字符串，后续很可能还要再拆一次
- 如果在 `run_agent.py` 中直接复制 step 循环，会加剧入口重复逻辑
- 如果在这一步就同时改 Web/CLI/交互入口，任务会迅速失控
- 如果不定义 `stop_reason`，后续入口层仍会各自解释执行状态

完成 `M1-T1` 后，应该至少得到以下产物：

- 一个可用的 CLI 主入口
- 一个 `AgentLoop` 高层运行接口
- 一个最小运行结果结构
- 一套主循环停止规则
- 一份最小验收场景说明

当前实际产物：

- `agent/custom_types.py` 中新增 `RunResult`
- `agent/loop.py` 中新增 `run_until_stop()`
- `scripts/run_agent.py` 已接入统一执行主干
- `scripts/chat_agent.py` 已接入统一执行主干，并恢复 step 级实时输出
- `scripts/openai_adapter.py` 非流式路径已接入统一执行主干
- `.gitignore` 已忽略 `sessions/`、`chats/`、`task.txt`

#### M1-T1 与后续任务的边界

`M1-T1` 完成后，后续任务再继续接：

- `M1-T2` 负责把这个高层接口扩展成通用单轮执行语义
- `M1-T3` 负责让其他入口复用它
- `M1-T4` 负责统一事件协议与 trace 渲染

也就是说，`M1-T1` 要做的是“补齐主干”，不是“一步重构完整个执行层”。

当前判断：

- `M1-T1` 已完成

验收依据：

遗留但不阻塞 `M1-T1` 完成的事项：

- `openai_adapter.py` 的流式路径仍保留旧的 step 循环
- 停止原因尚未细化到权限阻塞等更小粒度

#### 下一步工作

下一步进入 `M1-T2：统一 AgentLoop 执行语义`，重点做以下工作：

1. 收口 Web 流式路径

1. 先改 Web 流式路径
2. 再统一停止原因
3. 最后整理事件协议与对外接口命名

### M1-T2：统一 AgentLoop 执行语义

任务目标：
收口 `AgentLoop` 当前“部分方法返回事件对象、部分脚本拼接文本、部分入口自己驱动循环”的混杂状态。

当前问题：

- `step_once()` 返回 `List[AgentEvent]`
- `send_user_message()` 只执行一步，并不保证收敛到 final
- 不同入口对 “一轮对话” 的定义不一致

要实现的功能：

- 明确以下三个层级接口：
  - 单步接口：只执行一步，返回结构化事件
  - 单轮接口：处理一次用户输入，内部循环直到 final 或停止条件
  - 会话接口：负责启动、恢复、保存、摘要收尾
- 统一停止条件：
  - final
  - 达到 `max_steps`
  - 权限阻塞
  - 明确异常
- 统一最终结果数据结构，避免 Web/CLI 各自解释事件

建议输出：

- 一个标准的“单轮执行结果”结构，例如：
  - `final_text`
  - `trace_events`
  - `stop_reason`
  - `step_count`

完成标准：

- CLI 和 Web 都能复用同一套执行语义
- 不同入口不再各自维护一套终止判断
- 事件与最终答复之间的关系明确

### M1-T3：梳理三类入口职责边界

任务目标：
明确项目内三个入口的角色，避免逻辑重复和后续维护混乱。

涉及入口：

- `scripts/run_agent.py`
- `scripts/chat_agent.py`
- `scripts/openai_adapter.py`

要实现的功能：

- 为三个入口定义清晰定位：
  - `run_agent.py`：一次性任务执行入口
  - `chat_agent.py`：本地交互式调试入口
  - `openai_adapter.py`：Web / OpenWebUI 对接入口
- 抽取公共驱动逻辑，避免各入口重复处理：
  - agent 初始化
  - 单轮执行
  - 错误收敛
  - trace 输出转换
- 删除不必要的入口分叉逻辑或兼容性补丁

建议输出：

- 一个共享的驱动层，供三个入口复用
- 一份入口职责说明，方便后续扩展

完成标准：

- 三个入口不再重复维护相同流程
- 每个入口的输入输出职责清晰
- 新增入口时可以复用公共驱动层

### M1-T4：统一事件协议与可观察性

任务目标：
让系统的运行过程可观察、可调试、可被不同前端消费。

当前问题：

- 当前有 `AgentEvent`，但不同入口对事件的展示方式不一致
- Web 适配层仍包含较多事件重组和格式化逻辑

要实现的功能：

- 固化事件类型和字段约束，例如：
  - `thought`
  - `tool_call`
  - `tool_result`
  - `final`
  - `error`
- 明确每类事件的最小字段集
- 增加统一的事件转展示格式能力：
  - 纯文本
  - markdown
  - Web JSON
- 让 trace 的生成基于同一套事件源，而不是入口脚本二次推导

建议输出：

- 明确的事件协议文档或 dataclass 约束
- 标准 trace 渲染辅助函数

完成标准：

- CLI、交互式 CLI、Web 的 trace 内容一致
- 不同消费端只是渲染不同，不再重建业务语义
- 调试日志与用户可见 trace 可以区分

### M1-T5：补齐最小测试基线

任务目标：
为当前底座建立最小测试保护网，确保后续重构不会破坏已有功能。

测试范围建议：

- `agent/memory.py`
- `agent/chat_memory.py`
- `agent/loop.py` 的核心控制逻辑
- `tools/` 下的基础工具
- Web 适配层中的关键纯函数

要实现的功能：

- 为文件记忆读写补单元测试
- 为工具返回格式补单元测试
- 为 `AgentLoop` 的停止条件补测试
- 为 Web 层的 `final answer` 提取、conversation key 生成等纯函数补测试
- 对外部 LLM 调用做 mock，避免测试依赖真实网络

建议输出：

- 最小可运行测试集
- 本地测试执行命令说明

完成标准：

- 改动核心底座后可快速回归
- 测试不依赖真实 API key
- 至少覆盖主路径与关键边界条件

### M1-T6：收口权限策略

任务目标：
把当前临时关闭的权限逻辑恢复为可控的正式机制。

当前问题：

- `AgentLoop._maybe_request_permission()` 当前直接返回 `None`
- `AgentLoop._ask_user_for_permission()` 当前直接返回 `True`
- 权限系统存在实现，但主流程事实上未启用

要实现的功能：

- 明确当前阶段需要的权限策略：
  - 默认允许只读操作
  - 对路径修改命令进行检查
  - 对工作区外写入做阻断或显式确认
- 将 `PermissionManager` 真正接入执行流程
- 让权限阻塞成为标准停止原因之一
- 统一 CLI/Web 对权限阻塞的提示格式

建议输出：

- 可实际生效的权限判断链路
- 权限阻塞时的标准结构化结果

完成标准：

- 只读命令默认可执行
- 越界写入能被拦截
- 权限阻塞不会被误判为一般执行失败

### M1-T7：收口错误模型与停止原因

任务目标：
把当前“抛异常、打印日志、入口各自解释”的错误处理方式收口成统一模型。

当前问题：

- 有的错误直接 `raise RuntimeError`
- 有的入口层自己 catch 并拼接文本
- 停止原因缺少统一枚举

要实现的功能：

- 定义标准停止原因：
  - `final`
  - `max_steps`
  - `permission_blocked`
  - `tool_error`
  - `llm_error`
  - `invalid_model_output`
- 明确哪些错误属于“流程停止”，哪些属于“系统异常”
- 在单轮接口中统一返回 stop reason
- 让入口层只负责展示，不再自己判断业务含义

建议输出：

- 统一停止原因枚举
- 标准单轮返回结构

完成标准：

- 所有入口都能稳定识别停止原因
- 错误不再散落在各层自行解释
- Web 层不需要靠字符串特征判断核心状态

### M1-T8：整理配置与运行方式

任务目标：
清理当前配置项和运行方式中的不一致，降低使用成本。

当前问题：

- `config.json` 中已有多组配置，但入口之间使用方式不完全一致
- 用户不容易判断哪些配置是底座配置，哪些是入口配置

要实现的功能：

- 统一配置分层：
  - agent 基础配置
  - llm 配置
  - tools 配置
  - web 配置
  - script 入口配置
- 检查默认值是否合理
- 明确环境变量优先级
- 输出一份最小运行说明

建议输出：

- 整理后的配置说明
- 一份简单运行文档

完成标准：

- 新开发者可以快速理解如何运行项目
- CLI 与 Web 使用的配置来源清晰
- 缺失 API key、配置非法等情况有明确报错

## 5. 里程碑 1 内部执行顺序

建议里程碑 1 按以下顺序落地：

1. `M1-T1` 修入口断点
2. `M1-T2` 统一执行语义
3. `M1-T7` 收口停止原因与错误模型
4. `M1-T3` 梳理入口职责边界
5. `M1-T4` 统一事件协议与可观察性
6. `M1-T6` 收口权限策略
7. `M1-T8` 整理配置与运行方式
8. `M1-T5` 补齐测试并做一次整体回归

这样排序的原因：

- 先修掉主流程断裂，后续才有统一改造基础
- 先统一语义，再谈事件与前端展示
- 权限和配置应在主流程稳定后接入
- 测试最好覆盖收口后的最终形态
